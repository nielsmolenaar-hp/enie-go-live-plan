en-IE Go-Live — Technical Requirements (User-Story Overview)

A human-readable overview of what the new staging build must deliver for the Helloprint Ireland (en-IE) launch. Each requirement is written as a user story with commercial acceptance criteria and the business benefit first. The technical detail, acceptance criteria and Linear tickets sit in a collapsible "Technical detail & tickets" block under each one — there for context and engineering, but out of the way for a quick read.

Priority tiers: §A Must pass to launch (a build/CI gate blocks go-live if it fails) · §B Required for launch · §C After launch (deferred / V2).

Ticket status shorthand: ✅ Built — verify on staging · 🔧 Todo/in progress · ⛔ Blocked · 🆕 No ticket yet · ⚠️ Ticket conflicts with the agreed spec.

The commercial picture (read this first)

The en-IE launch puts ~6,754 product and category pages in front of Google and Irish customers. Everything below protects three commercial outcomes:

  • Get found — Google and AI search must be able to read, understand and index every page. A page they can't read earns no organic revenue.
  • Get the click — rich results (price, stars, breadcrumbs), correct country/language targeting, and fast pages win the click in the Irish search results.
  • Keep what we have — clean redirects, no dead links, and per-page index control protect existing rankings during the migration and stop thin/duplicate pages dragging the site down.

Where we stand today (from Linear): the product-page (PDP) foundation is essentially built and just needs verifying on en-IE staging. The category-page (PLP) build is mostly still open, and the sitemap, the canonical-vs-sitemap check, and the robots/bot-access work are not done yet. So the real launch work is: verify the PDP gates, finish the PLP build, ship the sitemap/robots/canonical layer, and fix a handful of open staging defects.

⚠️ Two tickets need re-scoping before the PLP schema is built: HEL-321 and HEL-326 still describe the old "CollectionPage + category star-rating" model. The agreed spec dropped both (category pages now use WebPage + product-list + breadcrumb, and there's no category-level star rating). Update these before building. (The related category-rating ticket HEL-330 is already cancelled — consistent.)

Open en-IE staging defects to fix as part of this work: JSON-LD template-token leak (HEL-1168), category-page links to a 404 (HEL-1139), contradictory filter-page prices (HEL-1145).

§A — Must pass to launch (deploy-block gates)

A1 · Google can read every page's content on load 🔴

User story. As a search engine (and AI assistant), I want a page's full content in the first response, so I can index it and rank it — and send Helloprint organic traffic. What "done" looks like (commercial): - Open any en-IE product or category page with JavaScript off — the title, description, price, reviews and links are all already there. - Nothing important appears only after a click or scroll. Benefit: a page Google can't read earns nothing. This is the floor under every page's ability to rank.

Technical detail & tickets - CI parser (cheerio/jsdom) asserts every SEO-bearing element is present + non-empty in the server HTML (H1, headings, breadcrumb + BreadcrumbList, intro, USPs, description, specs, FAQ Q&A incl. collapsed, SEO text incl. "show more", related tiles, JSON-LD, canonical/hreflang/robots, every ``, anchor text). Forbidden (build fails): `fetch()`-on-mount or client-render-only for indexable content; `display:none` on SEO-bearing elements; DOM removal. - **Tickets:** ✅ **HEL-285** SSR-output validators (deploy-blocking) *Done* — verify on en-IE. 🔧 PLP blocks: intro **HEL-327**, grid **HEL-328** *Todo*. ⚠️ open defect **HEL-1168** (token leak into JSON-LD). - **Owner:** ENG (Michael). **Source:** `mob-35/36`, g-mobile content parity.

A2 · Pages are fast enough to rank and convert 🔴 (PDP gated; PLP/FLP monitored)

User story. As an Irish mobile customer, I want pages to load quickly and not jump around, so I stay and buy — and as the business we keep Google's "good" speed rating that helps us rank. What "done" looks like (commercial): - Product pages pass Google's "good" Core Web Vitals on a mid-range phone; a slow page can't reach production. - Category pages are monitored to the same targets (not a hard gate yet). Benefit: speed is both a ranking factor and a conversion lever — slow mobile pages lose rank and sales.

Technical detail & tickets - **PDP (deploy-blocking, Lighthouse CI per PR, Moto G4/slow-4G):** LCP ≤2.0s · CLS ≤0.05 · INP ≤150ms · TTFB ≤250ms cached/600ms uncached · FCP ≤1.5s · JS ≤250KB gz · first-screen image ≤400KB · inlined CSS ≤14KB. **PLP (guideline, not gated):** same targets, JS ≤240KB gz, sampled weekly. **FLP (out of v1):** LCP ≤2.5 · INP ≤200 · CLS ≤0.1. - **Tickets:** ✅ **HEL-286** Lighthouse CI deploy-blocking gate *Done* — verify it runs on en-IE staging. (Homepage CWV fix **HEL-566** *Done* confirms the gate is live.) - **Owner:** Frontend (Frederico) + ENG. **Source:** `mob-19`, `mob-41`, g-mobile §3.16.

A4 · Each page is its own "original" (and we prove the link works) 🔴

User story.
